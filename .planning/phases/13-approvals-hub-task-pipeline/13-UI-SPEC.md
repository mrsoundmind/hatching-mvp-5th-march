## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge padding, color-accent bars |
| sm | 8px | Between approval buttons, between feed item elements |
| md | 16px | Card internal padding, tab content padding |
| lg | 24px | Between card groups, section separation |
| xl | 32px | Major section breaks in sidebar |
| 2xl | 48px | Not used at sidebar width |
| 3xl | 64px | Not used at sidebar width |

Exceptions:
- Touch targets for Approve Task/Reject Task buttons: minimum 36px height (accessibility; sidebar is narrow so full 44px not required — text + padding achieves 36px naturally at `px-3 py-1.5`)
- Pipeline stage column headers: `py-2` (8px) top/bottom — sidebar width constraint forces compact header rows
- Color-coded left border accent bar on approval items: 3px wide (matches existing `ActivityFeedItem` pattern)

Source: Established pattern in `ActivityFeedItem.tsx` and `AutonomousApprovalCard.tsx` — not re-specified, confirmed consistent.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (text-sm) | 400 (regular) | 1.5 |
| Label | 12px (text-xs) | 600 (semibold) | 1.4 |
| Micro | 10px (text-[10px]) | 400 (regular) | 1.3 |
| Heading | 16px (text-base) | 600 (semibold) | 1.2 |

Notes:
- Body (14px/400) is used for primary content in approval cards — agent name, task description.
- Label (12px/600) is used for status badges, pipeline stage labels, timestamps, button text. Weight raised from 500 to 600 — medium (500) at 12px is near-invisible as a weight distinction from regular at sidebar scale; semibold (600) provides clear hierarchy contrast.
- Micro (10px/400) is used for relative timestamps and secondary metadata — matches existing `ActivityFeedItem` pattern.
- Heading (16px/600) is used for empty state headings only; section titles inside the tab use Label.
- Only 2 weights in use: 400 (regular) and 600 (semibold).
- No markdown headers or bullet lists in any copy (CLAUDE.md prompt rule applies to components too).

Source: Confirmed from `ActivityFeedItem.tsx` (text-xs, text-[10px]) and `AutonomousApprovalCard.tsx` (text-sm font-semibold, text-xs). Applied consistently to Phase 13 components.

## Component Inventory

These components must be created in `client/src/components/sidebar/`:

### ApprovalsTab.tsx
Container for the full Approvals tab content. Renders `ApprovalsEmptyState` when no pending approvals; renders `ApprovalItem` list + `TaskPipelineView` when data exists.

### ApprovalItem.tsx
Single pending approval row within the Approvals tab. Renders:
- Agent avatar (32px circle, initial letter, same pattern as `ActivityFeedItem`)
- 3px color-coded left border (orange for high-risk, amber for mid-risk)
- Agent name (text-sm font-semibold, `var(--hatchin-text-bright)`)
- Risk reasons (text-xs, `var(--muted-foreground)`, joined with " · ")
- Approve Task button (hatchin-blue bg, white text, CheckCircle icon, 12px text)
- Reject Task button (red-500/30 border, red-400 text, XCircle icon, 12px text)
- Expiry indicator — when expired: replace buttons with "Expired" badge in red-500/20 bg, red-400 text

Layout: matches `AutonomousApprovalCard.tsx` anatomy — same visual language, different context (sidebar vs. chat). Do not introduce new visual vocabulary.

### TaskPipelineView.tsx
Horizontal pipeline showing task counts per stage. Renders 5 columns: Queued, Assigned, In Progress, Review, Done. At sidebar narrow width, renders as a vertical compact list of stage rows (not Kanban columns) — each row shows: stage name + task count badge.

Stage color system:
- Queued: text-muted-foreground, gray dot
- Assigned: hatchin-blue dot
- In Progress: hatchin-orange dot, pulsing animation
- Review: amber-400 dot
- Done: hatchin-green dot, no animation

### ApprovalsEmptyState.tsx
Empty state for the Approvals tab. Uses the existing `EmptyState` component from `client/src/components/ui/EmptyState.tsx` with `ShieldCheck` icon (already used in `SidebarTabBar`).

## Interaction Contracts

### Approve Task / Reject Task action
- Single click triggers action. No confirmation dialog for approve.
- Reject Task: no confirmation dialog (low-risk reversal — approvals are for paused autonomous tasks, not deletions).
- Both buttons enter `disabled + opacity-50` while the mutation is in flight (`isLoading: true`).
- On success: the `ApprovalItem` exits with Framer Motion `opacity: 0, y: -4` (200ms ease-out), matching `AutonomousApprovalCard` exit animation.
- On error: both buttons re-enable; show toast with "Couldn't process your decision. Try again." (no modal).

### Expired approval
- Expired items render with buttons replaced by a static "Expired" badge.
- No hover state, no click action.
- Expired items appear below active pending items (sorted: active first, expired last).
- Expired items are not counted in the Approvals tab badge.

### Task pipeline view
- Read-only. No drag-and-drop in this phase (deferred per REQUIREMENTS.md ADVN-03).
- Task counts are live — update when `task_created` or task status WS event is received.
- Clicking a stage row does nothing in this phase (no drill-down).

### Approvals tab badge
- The red dot on the SidebarTabBar `hasPendingApprovals` prop already exists in `SidebarTabBar.tsx`.
- Phase 13 must wire the actual count of non-expired pending approvals to this prop.
- Badge clears when tab is active and all visible approvals are actioned.

### Empty state interaction
- "Learn more" link is not needed — empty state is self-explanatory.
- No CTA button in empty state (approvals are created by the system, not the user).

## States Required

Each `ApprovalItem` must handle 4 states:

| State | Visual Treatment |
|-------|-----------------|
| Pending (default) | Full card with Approve Task/Reject Task buttons enabled |
| Loading | Approve Task/Reject Task buttons disabled, `opacity-50`, no spinner needed |
| Approved | Item exits via Framer Motion fade-out (removed from list) |
| Expired | Buttons replaced by "Expired" badge (red-400 text, red-500/10 bg) |

`TaskPipelineView` must handle 2 states:

| State | Visual Treatment |
|-------|-----------------|
| Loading | 5 skeleton rows at 24px height, `animate-shimmer` class (matches ActivityTab pattern) |
| Loaded | Stage name + count, colored dot per stage |

## Animation Contract

All animations use Framer Motion (established pattern across the codebase):

| Element | Animation |
|---------|-----------|
| ApprovalItem enter | `initial: {opacity:0, y:8}` → `animate: {opacity:1, y:0}` — 180ms ease-out |
| ApprovalItem exit | `exit: {opacity:0, y:-4}` — 200ms ease-out (matches AutonomousApprovalCard) |
| Pipeline stage row enter | `initial: {opacity:0}` → `animate: {opacity:1}` — 120ms ease-out, staggered 30ms per row |
| In Progress dot | CSS `animate-pulse` (Tailwind) — no Framer Motion needed for this |
| Pending approval dot on tab badge | CSS `animate-pulse` — already implemented in SidebarTabBar |

No spring physics for approvals — easeOut feels more decisive for an approval action.

## Accessibility Notes

- Approve Task and Reject Task buttons must have explicit `aria-label` including the task context: `aria-label={`Approve task: ${taskTitle}`}` and `aria-label={`Reject task: ${taskTitle}`}`.
- Pipeline stage list: use `role="list"` and `role="listitem"` — it is a structured enumeration.
- Approvals list: use `role="list"` — enables screen-reader traversal.
- Expired badge: `aria-label="Approval expired"` on the badge element.
- Empty state: `aria-live="polite"` on the container so screen readers announce when approvals clear.
