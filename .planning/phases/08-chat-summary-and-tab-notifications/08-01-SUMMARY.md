# Phase 8 Plan 1: Maya Return Briefing — Schema, Storage, and LLM Voice Summary

LLM-generated Maya briefing on user return after 15-minute absence using DB-backed presence tracking and idempotency guard.

## What Was Built

When a user returns to a project conversation after being away for 15+ minutes and autonomous work occurred, Maya now delivers one natural-language briefing message in the project chat. The briefing is:

- Generated by the LLM in Maya's voice, naming agents and leading with outcomes
- Stored as a real `agent` message persisted in chat history
- Broadcast as a `new_message` WS event (not the deprecated `return_briefing` event)
- Idempotency-guarded: `lastBriefedAt` is updated on success to prevent duplicates
- Silent when no autonomous work happened during absence
- Falls back to template text if LLM call fails

## Decisions Made

**DB-backed presence tracking:** `lastSeenAt` and `lastBriefedAt` added as nullable timestamp columns to `projects` table rather than tracking in memory. Survives server restarts, works correctly in multi-request scenarios.

**15-minute threshold:** Reduced from the hardcoded `Date.now() - 2 * 60 * 60 * 1000` (2 hours) to `ABSENCE_THRESHOLD_MS = 15 * 60 * 1000` (15 minutes) as specified by plan. More responsive to autonomous work that may happen quickly.

**LLM integration corrections (Rule 1 — Bug):** The plan specified `systemPrompt` field and `result.text`, but the actual `LLMRequest` type has no `systemPrompt` field (system prompt goes in `messages` as `role: 'system'`) and `LLMGenerationResult` uses `content` not `text`. Fixed inline during Task 3.

**idempotency pattern:** `lastSeenAt` is updated on every `join_conversation`. `lastBriefedAt` is only updated when `briefing.hasBriefing === true`. This ensures: first join triggers presence update, briefing only fires when 15+ min absent, second join immediately after does not re-brief.
